# Claude Code 시작하기

> **이 문서의 목표:** "Claude Code가 뭔지, 나는 뭘 하면 되는지" 10분 안에 파악하기

---

## 1. Claude Code는 일반 AI 챗봇이 아닙니다

ChatGPT나 Claude.ai 웹과의 **결정적 차이점**:

| 일반 AI 챗봇 | Claude Code |
|-------------|------------|
| 대화만 가능 | **파일을 직접 읽고, 쓰고, 실행**할 수 있음 |
| 코드를 복사-붙여넣기 | **프로젝트 구조를 이해**하고 여러 파일을 동시에 수정 |
| 매번 컨텍스트 설명 필요 | **CLAUDE.md를 매 세션 자동 로드** (프로젝트 규칙 기억) |
| 단일 응답 | **멀티에이전트** 시스템 (전문가 팀처럼 작동) |
| 테스트 실행 불가 | **터미널 명령어 실행** 가능 (빌드, 테스트, Git 등) |

**핵심:** Claude Code는 "대화 상대"가 아니라 **"프로젝트에 투입된 AI 개발팀"**입니다.

---

## 2. 당신의 역할: "사장님"

당신은 코드를 직접 쓸 수도 있지만, Claude Code 환경에서 가장 효과적인 역할은:

### 당신이 하는 것
- **목표 제시**: "음악 업로드 API 만들어줘"
- **승인/거부**: REQ(요구사항 정의서) 확인 후 "진행해" / "이건 바꿔"
- **방향 결정**: "MySQL로 가자", "이 라이브러리 쓰지 마"
- **결과 검수**: 완성된 코드/문서 확인

### 에이전트가 하는 것
- 요구사항 분석 및 문서화
- 아키텍처 설계
- 코드 작성/리팩토링
- 테스트 작성 및 실행
- 보안 검토
- 문서 정리

### 비유
```
당신 = 건축주 (뭘 짓고 싶은지 말하는 사람)
MA   = 현장 소장 (건축주 의도를 파악하고 전문가에게 분배)
에이전트들 = 전문 시공팀 (설계사, 구조기사, 보안 전문가...)
```

---

## 3. 대화의 기본 구조

### 가장 간단한 사용법

```
당신: "회원가입 API 만들어줘"
```

이 한 마디로 MA(메인 에이전트)가:
1. 의도를 파악하고 질문 (필요시)
2. REQ(요구사항 정의서) 초안 작성
3. 당신에게 승인 요청
4. 승인 후 작업 분할 및 실행
5. 결과 보고

### 더 좋은 사용법

```
당신: "회원가입 API를 만들어줘.
      - 이메일/비밀번호 기반 가입
      - 비밀번호는 BCrypt 암호화
      - 중복 이메일 체크 필요
      - 가입 성공 시 JWT 토큰 반환"
```

**구체적일수록 질문 왕복이 줄고 결과가 정확합니다.**

---

## 4. CLAUDE.md — 에이전트의 "헌법"

프로젝트 루트에 있는 `CLAUDE.md` 파일은 **매 세션마다 자동으로 로드**됩니다.

### 왜 중요한가?

| CLAUDE.md에 적으면 | 효과 |
|-------------------|------|
| 기술 스택 (Java 17, Spring Boot 4.x) | 에이전트가 해당 버전에 맞는 코드 작성 |
| 코딩 규칙 (Lombok 사용, DTO/Entity 분리) | 매번 말하지 않아도 규칙 준수 |
| 금지 사항 (Entity를 Controller에서 반환 금지) | 실수 사전 차단 |
| 패키지 구조 (config/, controller/, service/) | 파일을 올바른 위치에 생성 |

### 핵심 포인트

- CLAUDE.md를 수정하면 **다음 세션부터 즉시 반영**
- 여기에 적힌 규칙은 에이전트가 **반드시** 따름
- "항상 이렇게 해줘"라고 매번 말할 필요 없이, CLAUDE.md에 한 번 적으면 됨

## 5. 승인(Approval) 시스템 이해하기

Claude Code가 "멋대로" 하지 않도록 만든 안전장치입니다.

### 승인이 필요한 것

| 상황 | 왜 물어보나? |
|------|-----------|
| **REQ 승인** | "내가 이해한 게 맞나?" 확인 |
| **파일 삭제** | 돌이킬 수 없는 변경 |
| **DB 스키마 변경** | 데이터 영향 |
| **아키텍처 결정** | 장기적 영향 |
| **라이브러리 선정** | 기술 부채 가능성 |

### 자동으로 진행되는 것 (REQ 승인 후)

- Git 작업 (커밋, 브랜치)
- 빌드/테스트 실행
- 코드 생성/수정 (WI에 명시된 범위)
- 문서 생성

### 실전에서의 흐름

```
1. 당신: "로그인 기능 만들어줘"
2. MA: REQ 초안 작성 → "이렇게 이해했는데 맞나요?"
3. 당신: "오케이, 진행해" ← 이게 REQ 승인
4. MA: 자동으로 WI 분할 → 에이전트 위임 → 코드 작성 → 테스트
5. MA: "완료했습니다. 결과 확인해주세요."
```

## 6. 첫 번째 요청 해보기 (실전)

### Step 1: 그냥 말하기

```
"ATStudio에 음악 엔티티(Music)를 만들어줘.
제목, 아티스트명, 장르, 가격, 재생시간이 필요해."
```

### Step 2: REQ 확인

MA가 REQ 초안을 보여줍니다. 확인 포인트:
- 내가 말한 의도와 맞는가?
- 빠진 것은 없는가?
- 제약 조건이 합리적인가?

### Step 3: 승인

"좋아, 진행해" / "오케이" / "ㅇㅋ"

### Step 4: 결과 확인

- 생성된 파일 목록
- 코드 리뷰 (Entity, Repository, Service, Controller, DTO)
- 테스트 실행 결과

## 7. 자주 하는 실수 (처음에 꼭 읽기)

### 실수 1: 너무 모호하게 요청

```
❌ "뭔가 좋은 거 만들어줘"
✅ "음악 검색 API를 만들어줘. 제목, 아티스트, 장르로 검색 가능하게"
```

### 실수 2: 한 번에 너무 많이 요청

```
❌ "회원가입, 로그인, 음악 업로드, 결제, 정산 다 만들어줘"
✅ "먼저 회원가입 API부터 만들자" → 완료 후 → "다음은 로그인"
```

### 실수 3: REQ를 대충 승인

REQ는 **계약서**입니다. 꼼꼼히 읽고:
- 범위가 맞는지
- 빠진 게 없는지
- 제약 조건이 현실적인지

확인한 후 승인하세요.

### 실수 4: 승인 후 중간에 방향 전환

```
❌ 승인 후: "아 그냥 MySQL 말고 MongoDB로 바꿔"
✅ "잠깐, 범위 변경이 필요해. DB를 MongoDB로 바꾸고 싶어."
   → MA가 새로운 REQ 또는 변경 영향 분석을 진행
```
